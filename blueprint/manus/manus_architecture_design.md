# Growin Performance Test Framework: Architecture Enhancement Plan

## 1. User Authentication System

### Design Principles
*   **Security-first**: Prioritize secure credential handling and session management.
*   **Minimal Dependencies**: Leverage existing system capabilities where possible (e.g., SQLite).
*   **TUI Integration**: Seamlessly integrate authentication into the `pt-menu.sh` TUI.

### Architecture Proposal

#### Components:
1.  **`auth.py` (Python Module)**:
    *   Handles user creation, password hashing (bcrypt/argon2), and verification.
    *   Manages session generation, validation, and persistence.
    *   Interacts with an SQLite database for user and session storage.
    *   Provides functions for login, logout, session refresh, and force logout.
2.  **`auth_tui.sh` (Bash Script)**:
    *   Presents the login screen within the TUI.
    *   Captures username and password securely (e.g., using `read -s`).
    *   Calls `auth.py` for authentication and receives a session token.
    *   Stores the session token securely (e.g., in a temporary, permission-restricted file or environment variable for the current shell session).
3.  **SQLite Database (`auth.db`)**:
    *   Stores user credentials (hashed passwords), roles, and session information.
    *   Tables: `users`, `sessions`.

#### Authentication Flow:
1.  User launches `pt-menu.sh`.
2.  `pt-menu.sh` checks for an active, valid session. If none, it calls `auth_tui.sh`.
3.  `auth_tui.sh` prompts for username/password.
4.  `auth_tui.sh` passes credentials to `auth.py`.
5.  `auth.py` verifies credentials against `auth.db`, generates a session token, and records it.
6.  `auth.py` returns the session token to `auth_tui.sh`.
7.  `auth_tui.sh` stores the token and returns control to `pt-menu.sh`.
8.  `pt-menu.sh` proceeds with TUI display, using the session token for subsequent authorization checks.

#### Security Considerations:
*   **Password Hashing**: Use `bcrypt` or `argon2` via Python libraries.
*   **Session Persistence**: Store session tokens with limited lifespan and refresh mechanisms. Use file permissions to restrict access to session files.
*   **Local Credential Storage**: Avoid plaintext. Consider OS-level secure storage if available, or encrypted files with strict permissions.
*   **Audit Logging**: All login attempts (success/failure), session creations, and destructions will be logged.
*   **Concurrent Login Policy**: `auth.py` will enforce policies (e.g., single active session per user, or allow multiple but track them).

### Future Enhancements:
*   **LDAP/SSO Integration**: `auth.py` can be extended to support external authentication providers by abstracting the user verification logic.
*   **OAuth-ready**: Design the session token mechanism to be compatible with OAuth standards for future API integrations.

## 2. Role-Based Access Control (RBAC)

### Design Principles
*   **Hierarchical Roles**: Support a clear hierarchy of roles with configurable capabilities.
*   **Granular Permissions**: Control access to menus, commands, remote execution, and Docker operations.
*   **Extensibility**: Design for future integration with LDAP/SSO.

### Architecture Proposal

#### Components:
1.  **`rbac.py` (Python Module)**:
    *   Defines roles and their associated permissions.
    *   Provides functions to check user permissions for specific actions.
    *   Manages role inheritance and permission registry.
    *   Interacts with the SQLite database for role and permission storage.
2.  **`pt-menu.sh` (Modified Bash Script)**:
    *   Integrates with `rbac.py` to filter TUI menu visibility based on the logged-in user's role.
    *   Calls `rbac.py` before executing sensitive commands or remote operations.
3.  **SQLite Database (`auth.db` - extended)**:
    *   New tables: `roles`, `permissions`, `role_permissions`, `user_roles`.

#### RBAC Flow:
1.  After successful authentication, `pt-menu.sh` retrieves the user's roles from `auth.db` via `rbac.py`.
2.  `rbac.py` loads the permissions associated with these roles, including inherited permissions.
3.  When rendering the TUI, `pt-menu.sh` queries `rbac.py` to determine which menu items or options the user is authorized to see.
4.  Before executing any command (local or remote), `pt-menu.sh` makes an authorization check against `rbac.py`.
5.  `rbac.py` evaluates the user's permissions against the required permission for the action.

#### Role Examples (Configurable):
*   **god**: Full access, bypasses locks, emergency termination, impersonation.
*   **admin**: User management, role assignment, full execution rights.
*   **operator**: Execute PT, view dashboards.
*   **readonly**: Monitor status, view dashboards.
*   **guest**: Limited access.

#### Security Considerations:
*   **Centralized Permission Registry**: All permissions are defined and managed in `rbac.py` and stored in `auth.db`.
*   **Role Inheritance**: Roles can inherit permissions from parent roles, simplifying management.
*   **Audit Logging**: All authorization attempts (success/failure) and permission changes will be logged.

### Future Enhancements:
*   **LDAP/SSO Integration**: Map LDAP/SSO groups to internal roles within `rbac.py`.
*   **Dynamic Permissions**: Allow administrators to define custom permissions and assign them to roles via a management interface (e.g., a web UI or an extended TUI menu).

## 3. Concurrent Test Detection

### Design Principles
*   **Active Runtime Tracking**: Maintain a real-time registry of all active PT executions.
*   **Execution Locking**: Prevent concurrent executions on the same environment/server.
*   **Distributed Lock Mechanism**: Support locking across multiple servers and environments.

### Architecture Proposal

#### Components:
1.  **`runtime_registry.py` (Python Module)**:
    *   Manages the global execution state registry.
    *   Handles lock acquisition, release, and synchronization.
    *   Provides functions for heartbeat updates and stale lock cleanup.
    *   Interacts with the SQLite database for lock storage.
2.  **`pt-menu.sh` (Modified Bash Script)**:
    *   Integrates with `runtime_registry.py` to check for active locks before starting a test.
    *   Updates the status bar with real-time execution information.
    *   Provides options for queue mode, deny mode, and force takeover mode (for god role).
3.  **SQLite Database (`runtime.db`)**:
    *   Stores active executions, locks, and heartbeat information.
    *   Tables: `active_runs`, `locks`.

#### Concurrent Test Detection Flow:
1.  Before starting a test, `pt-menu.sh` queries `runtime_registry.py` to check for an active lock on the target environment/server.
2.  If a lock exists, `pt-menu.sh` prompts the user based on their role and the configured policy (e.g., deny, queue, force takeover).
3.  If no lock exists or the user chooses to force takeover, `pt-menu.sh` requests a lock from `runtime_registry.py`.
4.  `runtime_registry.py` creates a lock record in `runtime.db` and returns a lock ID.
5.  `pt-menu.sh` starts the test and periodically sends heartbeat updates to `runtime_registry.py` to maintain the lock.
6.  Upon test completion, `pt-menu.sh` releases the lock via `runtime_registry.py`.
7.  `runtime_registry.py` periodically checks for stale locks (locks without recent heartbeats) and cleans them up.

#### Status Bar Enhancements:
*   **Target Status Bar**: `IP: 10.100.201.192 │ ENV: INT │ VUs: 1 │ Dur: 30s │ Available Resource | Maul [Idle]`
*   **Active Execution Status Bar**: `IP: 10.100.201.192 │ ENV: INT │ VUs: 1 │ Dur: 30s │ Performance Test Ongoing | Script: BP120.js | ETA: 15m | By: Maul`

#### Security Considerations:
*   **Lock Synchronization**: Ensure locks are synchronized across multiple servers if the framework is distributed.
*   **Crash Recovery**: Implement mechanisms to recover from crashes and release stale locks.
*   **Execution Ownership**: Track the user who initiated the execution to prevent unauthorized lock releases.

### Future Enhancements:
*   **Queue Mode**: Implement a queueing system for tests when the environment is locked.
*   **Active User Dashboard**: Provide a real-time dashboard showing all active users and their executions.
*   **PT Occupancy Map**: Visualize the occupancy of different environments and servers.

## 4. Resource Management System

### Design Principles
*   **Real-time Resource Awareness**: Continuously monitor system resources on PT servers.
*   **Overload Prevention**: Proactively prevent test executions that would over-utilize resources.
*   **Dynamic Feedback**: Provide real-time resource status to users.

### Architecture Proposal

#### Components:
1.  **`resource_monitor.py` (Python Module)**:
    *   Collects real-time metrics for CPU, RAM, Docker containers, k6 processes, SSH sessions, running scripts, and concurrent VUs.
    *   Estimates bandwidth usage.
    *   Calculates a server health score and estimated free capacity.
    *   Provides safe execution recommendations based on current load and configured thresholds.
2.  **`pt-menu.sh` (Modified Bash Script)**:
    *   Integrates with `resource_monitor.py` to display dynamic status bar indicators (green/yellow/red).
    *   Uses resource data for smart scheduling and execution throttling.
3.  **SQLite Database (`runtime.db` - extended)**:
    *   New table: `server_registry` to store server health metrics and resource utilization history.

#### Resource Management Flow:
1.  `resource_monitor.py` runs as a background daemon or is invoked by `pt-menu.sh` to collect system metrics from local and remote PT servers.
2.  Metrics are stored in `server_registry` table in `runtime.db`.
3.  Before initiating a test, `pt-menu.sh` queries `resource_monitor.py` for the health and available capacity of the target server.
4.  `resource_monitor.py` provides a recommendation (e.g., go, throttle, deny) based on predefined concurrency thresholds and resource limits.
5.  The TUI status bar dynamically updates to reflect the server's health, occupancy percentage, and estimated free capacity.
6.  If a test is throttled or denied, the user receives appropriate feedback.

#### Dynamic Status Bar:
*   **Green/Yellow/Red State**: Visual indicator of server health.
*   **Occupancy %**: Percentage of resources currently in use.
*   **Estimated Free Capacity**: Remaining capacity for new tests.

#### Security Considerations:
*   **Secure Data Collection**: Ensure resource metrics are collected securely, especially from remote servers.
*   **Threshold Configuration**: Properly configure thresholds to avoid false positives or negatives.

### Future Enhancements:
*   **Smart Scheduling Engine**: Implement advanced algorithms for optimal test placement and resource allocation.
*   **Cooldown Mechanism**: Introduce a cooldown period after heavy load to allow systems to recover.
*   **Bandwidth Estimation**: Refine bandwidth estimation for more accurate resource planning.

## 5. User Management Menu

### Design Principles
*   **Administrative Control**: Provide a TUI-based interface for managing users and their roles.
*   **Security**: Ensure all user management operations are secure and auditable.
*   **Role-Based Access**: Restrict access to user management functions based on RBAC.

### Architecture Proposal

#### Components:
1.  **`user_manager.py` (Python Module)**:
    *   Provides functions for creating, deleting, and updating users.
    *   Handles password resets, role assignments, account locking/unlocking.
    *   Interacts with the SQLite database (`auth.db`) for user data.
    *   Integrates with `rbac.py` for authorization checks on management actions.
2.  **`user_menu.sh` (Bash Script)**:
    *   Presents the user management TUI menu.
    *   Calls `user_manager.py` to perform user-related operations.
    *   Ensures only authorized roles (e.g., `god`, `admin`) can access this menu.

#### User Management Flow:
1.  An authorized user (e.g., `admin`) navigates to the User Management menu in `pt-menu.sh`.
2.  `pt-menu.sh` calls `user_menu.sh`, which displays options like "Create User", "Delete User", "Assign Role", etc.
3.  When an option is selected, `user_menu.sh` prompts for necessary input (e.g., username, new password, role).
4.  `user_menu.sh` calls the appropriate function in `user_manager.py`.
5.  `user_manager.py` performs the requested operation, updating `auth.db` and logging the action.
6.  `user_manager.py` enforces RBAC rules, denying unauthorized actions.

#### God Role Capabilities:
*   **Full Access**: Unrestricted access to all system functions.
*   **Bypass Locks**: Can override execution locks.
*   **Emergency Terminate Execution**: Force-stop any running test.
*   **Impersonation Mode**: Temporarily assume another user's identity for debugging/support.
*   **Audit Visibility**: Full access to audit logs.

#### Security Considerations:
*   **Secure Implementation**: All user management functions must be robust against injection attacks and unauthorized access.
*   **SQLite Schema**: Design the `users` table with appropriate fields for account status (locked, active), last login, etc.
*   **Audit Logging**: All user management actions (creation, deletion, role changes, password resets, lock/unlock) must be logged.

### Future Enhancements:
*   **Migration Strategy**: Plan for migrating existing users (if any) to the new authentication system.
*   **Web UI**: Consider a web-based interface for user management for easier administration in the future.

## 6. Active Execution Observability

### Design Principles
*   **Live Visibility**: Provide real-time insights into active performance test executions.
*   **Low Overhead**: Ensure monitoring mechanisms do not significantly impact system performance.
*   **Terminal Dashboard**: Deliver a rich, interactive experience within the TUI.

### Architecture Proposal

#### Components:
1.  **`execution_monitor.py` (Python Module)**:
    *   Retrieves live execution data from the `active_runs` table in `runtime.db`.
    *   Aggregates and formats data for display in the TUI.
    *   Calculates real-time metrics like current VUs, RPS, error rates, and ETA.
    *   Monitors server resource usage for active tests by integrating with `resource_monitor.py`.
2.  **`tui_renderer.sh` (Bash Script)**:
    *   Presents a terminal dashboard with live-refreshing data.
    *   Uses `tput` and ANSI escape codes for dynamic layout and colorization.
    *   Integrates with `execution_monitor.py` to fetch and display data.
    *   Provides a "top-like" monitoring experience.
3.  **SQLite Database (`runtime.db` - extended)**:
    *   The `active_runs` table will store detailed information about each running test, including script name, executor, start time, current VUs, environment, and target.

#### Observability Flow:
1.  When a user selects the "Active Execution Monitor" from the `pt-menu.sh` TUI, it invokes `tui_renderer.sh`.
2.  `tui_renderer.sh` enters a loop, periodically calling `execution_monitor.py`.
3.  `execution_monitor.py` queries `runtime.db` to get the latest status of all active tests.
4.  For each active test, `execution_monitor.py` retrieves relevant metrics and potentially calls `resource_monitor.py` to get real-time server resource usage.
5.  `execution_monitor.py` returns the formatted data to `tui_renderer.sh`.
6.  `tui_renderer.sh` clears the screen and renders the terminal dashboard with the updated information, including colorized states for critical metrics.
7.  The dashboard will display: script name, executor, start time, ETA, current VUs, RPS, error rate, remote/local target, environment, and server resource usage.

#### Terminal Dashboard Features:
*   **Live Refresh**: Data updates every few seconds.
*   **Top-like Monitoring**: Displays a list of active tests with key metrics.
*   **Execution Timeline**: (Future) Visual representation of test progress.
*   **Colorized State**: Use colors (e.g., green for healthy, red for errors) to highlight important information.

### Future Enhancements:
*   **Websocket Optional Future Design**: Explore using websockets for more efficient real-time data push to a potential web-based dashboard.
*   **Low Overhead Architecture**: Continuously optimize data collection and rendering to minimize performance impact.
*   **Live PT Inspector**: Allow users to drill down into a specific running test for more detailed metrics and logs.

## 7. Audit Logging & Forensics

### Design Principles
*   **Enterprise Audit Logging**: Capture all significant actions within the system.
*   **Immutability**: Ensure logs are append-only and tamper-proof.
*   **Searchability**: Facilitate easy searching and analysis of log data.

### Architecture Proposal

#### Components:
1.  **`audit_logger.py` (Python Module)**:
    *   Provides a centralized logging interface for all system components.
    *   Writes audit events to the SQLite database (`audit.db`).
    *   Ensures logs are append-only.
2.  **`pt-menu.sh` and other Python Modules (Modified)**:
    *   All critical actions (login, logout, PT execution, scheduler actions, ENV modifications, Docker operations, permission changes, force kills, lock takeovers) will call `audit_logger.py` to record events.
3.  **SQLite Database (`audit.db`)**:
    *   Stores all audit logs.
    *   Table: `audit_logs`.

#### Audit Logging Flow:
1.  Any component performing a critical action (e.g., `auth.py` for login, `runtime_registry.py` for lock acquisition, `user_manager.py` for user changes) calls `audit_logger.py`.
2.  `audit_logger.py` records the event, including timestamp, user, action, target, and outcome, into the `audit_logs` table in `audit.db`.
3.  The `audit_logs` table is designed to be append-only, preventing modification of historical records.
4.  A separate TUI menu option or a Python script can be used to search and view audit logs.

#### Security Considerations:
*   **Immutable Logs**: Crucial for forensic analysis and compliance.
*   **Retention Policy**: Implement a clear policy for how long audit logs are retained.
*   **SIEM-ready**: Design the log format to be easily ingestible by Security Information and Event Management (SIEM) systems.
*   **Suspicious Activity Detection**: Implement rules or scripts to analyze audit logs for patterns indicative of brute-force attacks, lock abuse, or unauthorized access attempts.

### Future Enhancements:
*   **Centralized Log Management**: Integrate with external log management solutions (e.g., ELK stack, Splunk) for large-scale deployments.
*   **Alerting**: Configure alerts for suspicious activities detected in the audit logs.

## 8. Architecture Modernization

### Design Principles
*   **Modularization**: Break down the monolithic structure into smaller, manageable components.
*   **Separation of Concerns**: Clearly define responsibilities for Bash and Python components.
*   **Scalability**: Design for future growth and distributed orchestration.

### Architecture Proposal

#### Current State Analysis:
The existing `growin_performancetest` repository exhibits a pragmatic, yet tightly coupled, architecture. The `pt-menu.sh` script serves as a central orchestrator, handling TUI presentation, environment setup, execution logic, and even some basic reporting. While effective for a single-user or small team environment, this approach introduces several challenges for enterprise-grade multi-user scenarios:

*   **Tight Coupling**: Bash script mixes presentation, business logic, and infrastructure concerns.
*   **Limited Testability**: Bash scripts are inherently harder to unit test and debug compared to Python.
*   **Concurrency Hazards**: Lack of native concurrency primitives in Bash necessitates external locking mechanisms, which are currently absent.
*   **Maintainability**: Complex Bash logic can become difficult to maintain and extend.
*   **Scalability**: The current architecture is not well-suited for distributed execution or advanced orchestration.

#### Proposed Modernization Strategy:

We propose a **hybrid approach** that leverages the strengths of both Bash and Python, while providing a clear roadmap for future evolution.

1.  **Retain Bash TUI for Frontend**: The existing `pt-menu.sh` (and associated TUI scripts) will be refactored to primarily focus on user interaction, menu navigation, and displaying information. All complex business logic, data persistence, and external integrations will be delegated to Python modules.
2.  **Python for Core Logic and Services**: All new features (User Authentication, RBAC, Concurrent Test Detection, Resource Management, Audit Logging) will be implemented as Python modules. These modules will expose clear APIs (functions/classes) that can be called by the Bash TUI.
3.  **SQLite as Local/Shared State**: SQLite will serve as the primary data store for user management, roles, permissions, active runs, locks, and audit logs. This provides a lightweight, embedded database solution suitable for shared PT infrastructure.
4.  **Modularization and Plugin Architecture**: The Python codebase will be structured into distinct modules (e.g., `auth`, `rbac`, `runtime_registry`, `resource_monitor`, `audit_logger`). This promotes separation of concerns and allows for easier extension through a plugin architecture.
